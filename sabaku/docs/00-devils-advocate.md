# Devil's Advocate: SABAKU (捌く) — ビルドすべきか？

## Q1: 「Todoist / Linear / Notion Board で十分では？」

### 反論の核心
汎用タスク管理ツールはすでに成熟市場。SABAKUが解決する「問題」は既存ツールで対応可能に見える。

### SABAKUの不可替性

| 観点 | 汎用ツール | SABAKU |
|------|-----------|--------|
| データ入力 | 手動入力が前提 | Obsidian Vault から自動パース。追加作業ゼロ |
| タスク生成 | ユーザーが作成 | /daily, /log, /routing の既存ワークフローから自然発生 |
| フォーカスモデル | マルチタスク許容 | ACTIVE 1枚制限。管制官メタファーで「次の1つ」に集中 |
| 時間記録 | 別ツール（Toggl等）が必要 | Strip に内蔵タイマー。作業 → 記録が一体化 |
| 認知負荷 | アプリ間コンテキストスイッチ | Vault が Single Source of Truth。SABAKU は表示レイヤー |

**結論**: 汎用ツールは「タスクを管理する」ツール。SABAKUは「既にVaultで生まれたタスクを捌く」ビュー。入力コストゼロが最大の差別化。

## Q2: 「Obsidian連携が複雑すぎてユーザーが離脱するのでは？」

### 反論の核心
Obsidian Vault のパス構造、Markdown フォーマット、Claude コマンド体系に依存する。一般ユーザーには壁が高い。

### オンボーディング設計

**ターゲットの明確化（重要）**:
SABAKUのターゲットは「全Obsidianユーザー」ではない。

- **Primary**: 既に Obsidian で daily journal + checkbox タスク管理をしている人
- **Secondary**: Obsidian Tasks / Kanban プラグインに不満を持つパワーユーザー
- **Not target**: Notion / Todoist で満足している人、Obsidian 初心者

**セットアップフロー**:
1. Vault パスを指定（1ステップ）
2. 自動スキャン → 検出されたパターンをプレビュー表示
3. 「この3件のTODOをストリップにしますか？」→ 確認 → 同期開始

**複雑さの緩和策**:
- CLI の `sabaku sync --dry-run` でプレビュー可能
- デフォルト設定で `/daily` + `/log` の標準フォーマットに対応
- カスタムパースルールは後から追加可能（Progressive Disclosure）

**リスク判定**: 中。ターゲットを絞れば致命的ではないが、TAM（Total Addressable Market）は限定的。

---

## Q3: 「シングルタスク強制はパワーユーザーに嫌われるのでは？」

### 反論の核心
ACTIVE 1枚制限は制約が強すぎる。並行タスクが必要な場面は多い。

### ターゲット明確化

**SABAKUが解決するペルソナ**:
- ADHD-PI / HSP傾向：刺激に弱く、マルチタスクで消耗する
- WMI 平均域：ワーキングメモリに余裕がなく、1つずつ処理が最適
- Deep Work 実践者：Cal Newport 的「集中の深さ × 時間 = 成果」を信じる人

**設計上の回答**:
- ACTIVE 1枚は「推奨」であり、UX上の強い誘導（警告表示）で実現
- 技術的にはACTIVE 2枚も可能（将来のPro設定として）
- QUEUE にストリップを並べて「次にやること」を可視化 → マルチタスク感は担保

**リスク判定**: 低。シングルフォーカスはコア哲学。合わない人は最初からターゲットではない。

## Q4: 「フライトストリップUIは見た目の新鮮さだけで実用性が低いのでは？」

### 反論の核心
メタファーに引っ張られて情報密度が犠牲になる可能性。

### 情報密度の最適化

**フライトストリップの本質的利点**:
- 横長カード → 一覧性が高い（Kanban の縦長カードより情報が入る）
- 実際のATC strip は「一目で必要情報が分かる」ために設計されている
- 紙の短冊 → 物理的操作感（ドラッグ&ドロップとの親和性）

**SABAKUストリップの情報設計**:
```
[FS-0042] [URG] KASHITE Resend API 設定  [daily-top3] [⏱ 00:23:45] [🔗 Vault]
```
- ID: 一意識別（4桁で十分）
- Priority バッジ: 色で即認識
- タイトル: 本文
- カテゴリ: ソース元の文脈
- タイマー: ACTIVE 時のみ表示
- Source: Vault 自動 or 手動

**Kanban カード比較**:
- Notion Board: タイトル + 小さいタグ。密度低い
- Linear: 開発者向けで重い。個人タスクには過剰
- SABAKU Strip: 1行に全情報。スキャンが速い

**リスク判定**: 低。メタファーに忠実すぎない。実用性ファースト、ATC はフレーバー。

## Q5: 「既に Flightstrip (by nephros) が存在する」

### 調査結果

[Flightstrip on Codeberg](https://codeberg.org/nephros/Flightstrip) は ATC インスパイアのタスク管理ツール。

**Flightstrip の特徴**:
- スケジュールボード方式（カレンダーとTODO統合）
- タスクは動的、タイムスロットは静的という前提
- デスクトップアプリ（Qt/QML ベース）

**SABAKUとの差別化**:

| 観点 | Flightstrip (nephros) | SABAKU |
|------|----------------------|--------|
| プラットフォーム | デスクトップ（Qt） | Web（Next.js）+ PWA |
| データソース | 手動入力 | Obsidian Vault 自動連携 |
| フォーカスモデル | スケジュールボード | シングルフォーカス + Kanban |
| タイマー | なし | 内蔵ポモドーロ風 |
| デザイン | ATC緑モノクロ | ダークモダン + マルチカラー |
| ターゲット | 汎用 | Obsidian パワーユーザー |

**結論**: 名前とメタファーは近いが、プロダクトカテゴリが違う。SABAKU は「Obsidian Vault のタスクビューア」であり、汎用タスク管理ではない。

---

## 最終結論: ビルドすべきか？

### ビルドすべき理由
1. **ゼロ入力コスト**: 既存 Vault ワークフロー（/daily, /log, /routing）から自動生成。類似なし
2. **Dog-fooding 可能**: オペレーター自身が毎日使う。Pain が明確
3. **Kyrenラインの補完**: KASHITE（貸借） / YOMU（読書） / Phrasely（英語） に対して SABAKU（タスク）は自然な拡張
4. **TAM は小さいが深い**: Obsidian パワーユーザー × タスク管理不満層。ニッチだが熱量が高い

### スコープ調整の提案
- **V1**: Web UI + CLI sync のみ。Obsidian プラグインは V2 以降
- **V1**: 自分の Vault フォーマットにハードコード。汎用パーサーは V2
- **V1**: シングルユーザー。マルチユーザーは V2
- **Gumroad 販売**: CLI sync ツール単体でも価値がある（Vault → Supabase pipeline）

### リスクマトリクス

| リスク | 影響 | 確率 | 対策 |
|--------|------|------|------|
| TAM が小さすぎる | 中 | 高 | Dog-food + OSS で認知。Gumroad で validation |
| Vault フォーマット変更 | 高 | 低 | パーサーを疎結合に。config でルール定義 |
| 既存ツールで十分 | 中 | 中 | 「ゼロ入力」UX で差別化。使い始めのハードルを下げる |
| 開発コスト過大 | 中 | 低 | Kyren 共通パターン流用。2-3日で MVP |

**判定: GO（スコープ調整版で）**
